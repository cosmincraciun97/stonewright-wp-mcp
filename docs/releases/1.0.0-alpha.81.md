# 1.0.0-alpha.81

## Highlights

Critical self-improvement and write-safety repair after a live incident where
`php-execute` could append invalid PHP to theme `functions.php` while audit
recorded only snippet return success.

- **`php-execute` never writes code files.** Filesystem mutation APIs are
  permanently blocked (`stonewright_php_code_file_write_blocked`). Use
  `stonewright/theme-file-patch`.
- **Transactional theme-file writes.** Complete-candidate PHP/CSS/JS validation,
  custom-code operator grant, atomic replace, readback, bootstrap smoke, and
  automatic rollback.
- **Custom-code operator grant.** Single-use, short-lived, bound to path +
  candidate `after_sha256`; issued only from authenticated wp-admin.
- **Direct memory scope honesty.** Unknown site aliases fail with
  `site_alias_unresolved` (no silent `_global` write). Receipts label
  local-only visibility.
- **Effect-oriented audit metadata.** Execution / verification / rollback
  statuses and resource refs for mutations.
- **Safety blocks ≠ learning spam.** Expected blocks do not promote active
  project/user rules; audit feedback stays unresolved until verified repair.
- **Production mode P0 warning** when WordPress environment is production but
  Stonewright mode is not `production-safe`.
- **Canonical rule** `custom_code_operator_grant` (Plugin / Direct / skill parity).

## Upgrade notes

- Update the plugin and companion together to `1.0.0-alpha.81`.
- Restart or reload the MCP client after replacing the companion package.
- On production hosts, switch Stonewright mode to `production-safe` before
  agent writes.
- Theme PHP/CSS/JS apply requires dry_run + custom-code grant; do not retry
  code-file writes through `php-execute`.
- Direct learning for an unknown site alias now errors instead of writing
  `_global.jsonl`.
