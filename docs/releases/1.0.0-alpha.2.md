# Stonewright 1.0.0-alpha.2 Release Notes

**Released:** 2026-05-22
**Plugin package:** `stonewright/wp-mcp`
**Version string:** `1.0.0-alpha.2`

---

## What's new

See [CHANGELOG.md](../../plugin/CHANGELOG.md) for the full diff from alpha.1.

Summary of major additions in this release:

- **Elementor V3 renderer completeness** — native container/widget output for all core spec node types; diagnostics for every unsupported node; responsive settings preserved per viewport.
- **Elementor Widget Builder** — `stonewright/elementor-widget-*` abilities create, audit, register, preview, and delete editable V3 widget projects under `wp-content/stonewright-sandbox/widgets/`.
- **Companion QA contract** — real screenshot, pixel-diff, axe, layout, and Lighthouse endpoints with typed request/response contracts validated by `CompanionContract`; `QaArtifactStore` stores artifacts under `wp-content/uploads/stonewright-qa/`.
- **Security hardening** (Phases 1–3) — `ConfirmationToken` now signs over a canonical args hash, replay protection via transients, and a real TTL. `StaticGuard` blocks dynamic dispatch, variable functions, and `call_user_func` with unsafe targets. Sandbox actions require `edit_plugins + manage_options`.
- **Content capability checks** — `CreatePost`, `UpdatePost`, `BulkCreate` derive caps from `get_post_type_object()` and enforce `create_posts` / `publish_posts` per type.
- **Gutenberg / FSE parity** — `columns`, `cover`, `media-text`, `list`, `quote`, `buttons`, `group`, `spacer`, `image`, `video`, `embed`; `theme.json` global styles write requires a confirmation token in production-safe mode.
- **Memory and System abilities** — `stonewright/memory-*`, `stonewright/instructions-*`, `stonewright/abilities-list`.
- **Design ingestion** — `FigmaImporter` section-index fix; `NormalizeAssets` sidelo‌ads image fills to WordPress media; `BuildSpecFromInstructions` validates agent-produced specs.
- **Ability truth matrix** — generated by `composer docs:matrix` to `docs/ability-truth-matrix.md`.
- **Phase 10 smoke test** — `plugin/tests/Integration/SmokeTest.php` asserts all 7 hard rules and exercises every registered ability's interface without fatals.
- **Security audit script** — `plugin/bin/security-audit.php` (runnable via `composer security:audit`).

**Ability count:** 108 registered.

None relative to 1.0.0-alpha.1. The Ability interface, JSON schema format, and REST endpoint namespace are unchanged.

## Known limitations

The following are documented limitations that will be addressed before 1.0.0 stable:

1. **Elementor V4 is experimental.** `stonewright/elementor-v4-*` abilities require the `stonewright_elementor_v4_atomic` option to be enabled and are blocked in `production-safe` mode. Output is not guaranteed to load in the Elementor V4 editor without a live Elementor installation.
2. **Sandbox PHP execution risk.** Even with `StaticGuard`, executing user-supplied PHP is inherently dangerous. Sandbox activation requires `edit_plugins + manage_options` and a confirmation token in production-safe mode, but should not be exposed to untrusted editors.
3. **Figma ingestion stubs.** `ImportFigmaNode` and `IngestFigma` require a live Figma API token and companion connection. Without those, they return validated empty specs or structured errors.
4. **No real Playwright execution in tests.** The companion HTTP contract is stubbed in PHPUnit. Real screenshot/diff artifacts require a running companion with Node 22 and a browser.
5. **`ApplyFixPlan` is not yet fully implemented.** It returns a structured `not_implemented` error rather than silently claiming success.
6. **Elementor theme builder.** Template location assignment (`assign-template-location`) and dynamic tag validation (`validate-dynamic-tags`) require Elementor Pro and will fail gracefully on free installs.
7. **`optimize-media` is a stub.** Without a server-side image library, the ability returns the original attachment URL.

## Upgrade path from alpha.1

1. Deactivate the alpha.1 plugin.
2. Upload/replace the plugin directory with alpha.2 files.
3. Reactivate. No database migration is required.
4. The `stonewright_confirmation_secret` option is created lazily on first token issue — nothing to migrate.
5. If you have custom code referencing `ConfirmationToken::verify( $token, $ability, $args )`, the signature is unchanged. The payload format changed (added `args_hash`, real TTL), so any alpha.1 tokens will fail verification after upgrade — this is expected and safe.
